## About This Repo

This is a fork of [DIYgod/RSSHub](https://github.com/DIYgod/RSSHub) — an open-source RSS feed generator. This fork adds custom routes for Sunbi (a Korean-content browser extension) on top of the upstream codebase.

Custom Sunbi routes live in `lib/routes/` under their own namespaces (e.g. `sunbi-youtube/`). Upstream routes are not modified to avoid merge conflicts when pulling from upstream.

## Commands

```bash
# Dev server (hot reload, localhost:1200)
pnpm dev

# Run all tests (format check + vitest coverage)
pnpm test

# Run vitest only (faster)
pnpm vitest

# Run a single test file
pnpm vitest lib/middleware/cache.test.ts

# Run tests in watch mode
pnpm vitest:watch

# Lint
pnpm lint

# Format (auto-fix)
pnpm format

# Build routes manifest (required before building)
pnpm build:routes

# Full build
pnpm build
```

Node ≥22.20 required. Package manager: `pnpm@10`.

## Architecture

### Request Pipeline

`lib/index.ts` → `lib/app.ts` → `lib/app-bootstrap.tsx`

The Hono app applies middleware in order: logger → trace → honeybadger/sentry → access-control → debug → template → header → anti-hotlink → parameter → cache → routes.

Routes are registered via `lib/registry` (auto-generated by `build:routes`). Each route namespace lives in `lib/routes/<namespace>/` and exports a `namespace.ts` and one or more route files.

### Route Structure

Each route file exports a `route: Route` constant:

```typescript
import type { Route } from '@/types';

export const route: Route = {
    path: '/channel/:id',
    categories: ['social-media'],
    example: '/youtube/channel/UCxxx',
    parameters: { id: 'Channel ID' },
    radar: [{ source: ['www.youtube.com/channel/:id'], target: '/channel/:id' }],
    name: 'Channel',
    maintainers: ['handle'],
    handler,
    features: {
        requireConfig: [],
        requirePuppeteer: false,
        antiCrawler: false,
        supportBT: false,
        supportPodcast: false,
        supportScihub: false,
    },
};

async function handler(ctx) {
    // return { title, link, item: DataItem[] }
}
```

Key types are in `lib/types.ts`: `Route`, `DataItem`, `Namespace`, `Category`.

### HTTP Clients

- **`lib/utils/ofetch.ts`** — primary HTTP client (wraps `ofetch`). Use for most requests.
- **`lib/utils/got.ts`** — `getFakeGot()` compatibility shim that adapts `got`-style API to `ofetch`. Many older routes use this; prefer `ofetch` directly in new routes.

Both clients share proxy support, retry logic, and cookie jar handling configured centrally.

### Caching

`lib/middleware/cache.ts` handles response-level caching. Within route handlers, use `cache.tryGet(key, fetcher, ttl)` for fine-grained caching of individual fetched resources (e.g. per-episode data).

### Custom Sunbi Routes

Routes added for Sunbi are documented in `docs/impl/` (see [`docs/impl/README.md`](docs/impl/README.md)):

| Doc                        | Route                           |
| -------------------------- | ------------------------------- |
| `IMPL-01-spec-youtube.md`  | `/spec/youtube/:channelId`      |
| `IMPL-02-viki.md`          | `/spec/viki/:titleId`           |
| `IMPL-03-weverse.md`       | `/spec/weverse/:artistId`       |
| `IMPL-04-bubble.md`        | `/spec/bubble/:artistId`        |
| `IMPL-05-netflix.md`       | `/spec/netflix/:netflixTitleId` |
| `IMPL-08-naver-webtoon.md` | `/spec/naver/webtoon/:titleId`  |
| `IMPL-09-naver-blog.md`    | `/spec/naver/blog/:blogId`      |

All Sunbi routes attach structured metadata on each `DataItem` using the **`_extra`** field (see `lib/types/spec-extra.ts`). Payloads are discriminated with a `type` field (e.g. `youtube/video`, `naver/webtoon/episode`). JSON feeds expose this as `_extra`; this feeds downstream Sunbi ingestion. Do not use a bare `extra` property — it is not on `DataItem`.

### Deployment Targets

The repo supports three deployment modes:

1. **Node.js** — `pnpm start` (production), `pnpm dev` (dev)
2. **Cloudflare Worker** — `pnpm worker-build && pnpm worker-deploy`
3. **Cloudflare Container Worker** — `lib/container.ts` manages the RSSHub container lifecycle via Durable Objects, load-balanced across 20 instances

### Environment / Config

`lib/config.ts` reads all config from environment variables. Key variables for Sunbi:

| Var              | Purpose                                               |
| ---------------- | ----------------------------------------------------- |
| `ACCESS_KEY`     | Shared secret; all requests must include `?key=VALUE` |
| `CACHE_TYPE`     | `redis` in production, memory by default              |
| `REDIS_URL`      | Redis connection string                               |
| `WEVERSE_TOKEN`  | Bearer token (expires ~30 days)                       |
| `NETFLIX_COOKIE` | Full cookie header string                             |
| `NAVER_COOKIE`   | Cookie for paywalled content                          |

See `docs/routes/RSSHUB_SETUP.md` for Docker Compose setup and full environment reference.

### Testing

Tests use Vitest. Coverage is collected for `lib/**` excluding routes. Route tests use `msw` for HTTP mocking. The `lib/setup.test.ts` file runs before all tests.

To test a running instance manually:

```bash
curl "http://localhost:1200/<namespace>/<route>?format=json"
```

## Active SPEC Routes

Custom Sunbi media routes live under `lib/routes/spec/`:

- `lib/routes/spec/youtube.ts` — `/spec/youtube/:channelId` (YouTube RSS, 30 min TTL)
- `lib/routes/spec/netflix.ts` — `/spec/netflix/:titleId` (TMDB bridge, 60 min TTL)
- `lib/routes/spec/viki.ts` — `/spec/viki/:titleId` (public REST, 60 min TTL)
- `lib/routes/spec/weverse.ts` — `/spec/weverse/:artistId` (HMAC/Bearer, 3 min TTL)
- `lib/routes/spec/bubble.ts` — `/spec/bubble/:artistId` (session cookie, 2 min TTL)

`SpecExtra` typed contract: [`lib/types/spec-extra.ts`](lib/types/spec-extra.ts). Production launch: [`docs/LAUNCH_RUNBOOK.md`](docs/LAUNCH_RUNBOOK.md).
